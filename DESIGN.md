# Design System: YunTool (雲科工具箱)

## 1. Overview

**Creative North Star: "The Campus Pocket Knife (校園隨身工具刀)"**

YunTool is the tool a student pulls out between classes to check one thing — a grade, tomorrow's schedule, how many credits are left — and puts away. Everything about the visual system serves that draw-and-return motion: familiar Material 3 surfaces the student already knows how to read, a single friendly typeface, and colour used to mean something (a status, a warning, the brand) rather than to decorate. The system rejects the official portal's cramped form-heavy density, the no-identity template look, and any flashy showpiece treatment that gets between the student and the data.

There is one deliberate tension worth naming: the personality is **efficient and no-nonsense**, but the typeface is **JFOpenHuninn (粉圓體)**, a warm, rounded humanist sans. That roundness — echoed in the corner radii and the flat, softly-tinted surfaces — is what keeps a functional utility from feeling clinical or bureaucratic. Warmth comes from the shapes and the type, never from decoration. The whole palette is generated by Material 3 from a single teal seed, so the app reads as one coherent surface rather than a stitched-together set of screens.

**Key Characteristics:**
- One teal seed (`#009688`) → the entire Material 3 role palette. No hand-picked accents competing with it.
- One typeface (JFOpenHuninn) carries every role. Hierarchy comes from **size and colour**, not weight.
- Flat surfaces with Material 3 tonal layering; shadows are rare and purposeful.
- Colour is semantic — brand, status, warning — never ornamental.
- A deliberate corner-radius scale (8 → 20px), matched to element size.

## 2. Colors

A single teal seed generates the working palette; a small set of committed colours carry brand, status, and the one signature dark surface.

### Primary
- **YunTech Teal** (`#009688`): The seed passed to `ColorScheme.fromSeed`. Material 3 derives `primary`, `secondary`, `surface`, `onSurface`, and every other role tonally from this one value. It surfaces directly as the primary-action colour (the success snackbar, the install button, selected states) and as the tint under every neutral surface.

### Secondary
- **Campus Amber** (`#FFC107`): The one hand-placed accent, used for highlight and attention that isn't an error — calendar highlights, the profile "extra info" chips, exam markers. Amber earns its place because it reads as "note this" without the alarm of red.

### Neutral
- **Portal Slate** (`#1E293B`) with **Slate Ink** (`#FFFFFF`): The single committed non-generated surface — the profile identity card. A deliberate dark anchor in an otherwise light, teal-tinted UI. Centralized as `_profileCardBg` / `_profileCardFg`; it is the one place a fixed dark surface is allowed.
- **Material 3 neutrals**: `surface`, `surfaceContainerHighest`, `onSurface`, `onSurfaceVariant`, `outline` — all tonally generated from the teal seed. Cards, list rows, dividers, and body text use these, never raw greys.

### Status
- **Pass Green** (`#16A34A`): Positive / present / completed status (attendance, credential token present).
- **Caution Amber** (`#D97706`): Warning / partial / pending status.
- **Error** (Material 3 `error` role): Failure and destructive emphasis. The logout action and error snackbars use it.

### Named Rules
**The One Seed Rule.** The palette comes from a single teal seed through Material 3. Never introduce a second hand-picked brand colour to "spice up" a screen; if a surface needs colour, it comes from a Material role or from the semantic status set.

**The Colour-Means-Something Rule.** Green, amber, and red are reserved for status and are never decorative. If a colour on screen doesn't encode a state or the brand, it shouldn't be there.

## 3. Typography

**Display / Body / Label Font:** JFOpenHuninn — 粉圓體 (with `sans-serif` fallback)

**Character:** One warm, rounded humanist sans covers the entire interface — headings, titles, body, labels, and data. It renders Latin and CJK from one file, so Chinese and English sit together without a jarring switch. Because the family effectively ships a single weight, **hierarchy is built from size and colour, not from weight contrast** — a bold heading and body text are the same strokes at different scales.

### Hierarchy
- **Headline** (24px, `headlineSmall`): Screen-level titles and the profile name. The largest type in the app; there is no oversized hero display.
- **Title** (16px, `titleMedium`): Section headers, list-row titles, dialog titles.
- **Body** (14px, `bodyMedium`): The default — list content, descriptions, most on-screen text.
- **Label** (12px, `bodySmall` / `labelSmall`): Secondary info, chips, the reference disclaimer, timestamps.

### Named Rules
**The Size-and-Colour Rule.** Emphasis is carried by scale and by moving toward the ink end of the neutral ramp — not by weight. Don't lean on faux-bold to signal importance; the single-weight font makes it thin and uneven.

**The One-Family Rule.** JFOpenHuninn is the whole type system. Never pair it with a second display or mono face; a product UI this focused doesn't need one.

## 4. Elevation

The system is **flat by default and leans on Material 3 tonal layering** — surfaces are distinguished by tone (`surface` vs `surfaceContainerHighest`), not by shadow. Depth is a tint, not a drop shadow. This keeps the UI calm and fast to scan, and it's the default that most cards, list surfaces, and sheets follow.

### Shadow Vocabulary
- **Anchor shadow** (`box-shadow: 0 * 0 rgba(0,0,0,0.15)`, soft): Reserved for the profile identity card and the floating bottom-nav pill — the two elements that are meant to sit *above* the page. Shadow here signals "this floats", not decoration.
- **Loading shimmer**: Skeleton surfaces (`shimmer_box`) stand in for content while it loads, instead of a spinner in empty space.

### Named Rules
**The Flat-Unless-Floating Rule.** Surfaces are flat and tonal at rest. A shadow appears only when an element genuinely floats over the content beneath it (the identity card, the nav pill). If it isn't floating, it has no shadow.

## 5. Components

### Buttons
- **Shape:** Pill / large radius (`{rounded.pill}`, 20px) for the primary filled button; standard Material 3 elsewhere.
- **Primary:** `FilledButton`, teal background (`primary`) with white text. Used for the single main action on a surface (e.g. the PWA install button, now on `secondaryContainer` where it's a softer secondary affordance).
- **Hover / Focus / Pressed:** Material 3 state layers — no custom treatment needed.

### Cards / Containers
- **Corner Style:** `{rounded.md}` (12px) for standard cards; `{rounded.lg}` (16px) for large feature cards (the profile identity card, graduation summary).
- **Background:** Material 3 `surfaceContainerHighest` for settings/content cards; **Portal Slate** for the one profile identity card.
- **Shadow Strategy:** Flat by default (see Elevation). Only the profile card floats.
- **Internal Padding:** `{spacing.md}`–`{spacing.lg}` (16–24px).

### List Items
- **Style:** Material 3 `ListTile` — leading icon, text label, trailing chevron or control. This is the backbone pattern for settings and most detail rows.
- **Rule:** Icon **and** text label always; never icon-only rows.

### Chips
- **Info chip:** Rounded (`{rounded.sm}`, 8px), amber icon + text on a faint tonal fill. Used for the profile "extra info" fields and calendar markers.

### Navigation
- **Floating pill bottom nav:** A custom five-tab bar; a rounded "pill" indicator slides behind the active tab (`AnimatedPositioned`). Floats above content with the offline banner rendered just above it. This is the app's signature component — the one place motion and elevation are spent deliberately.

### Feedback — Top Snackbar
- **Custom typed snackbar** (`SnackBarType.success | error | info | warning`), sliding from the top with a leading icon. Colour maps semantically: success → teal `primary`, error → `error`, warning → deep orange, info → `secondary`. Consistent, non-blocking feedback for every action.

### Named Rules
**The Radius-by-Size Rule.** Corner radius scales with element size — chips 8px, cards 12px, feature cards 16px, pills 20px — never one radius on everything. This scale is what keeps the rounded look deliberate rather than a default-Material reflex.

## 6. Do's and Don'ts

### Do:
- **Do** derive every colour from the teal seed or the semantic status set. One seed, one palette.
- **Do** build hierarchy from **size and colour**, since JFOpenHuninn is effectively single-weight.
- **Do** keep surfaces flat and tonal; spend a shadow only on something that genuinely floats.
- **Do** follow the radius scale (8 / 12 / 16 / 20px) by element size, and keep it consistent screen to screen — this is what makes the rounded aesthetic read as intent.
- **Do** always pair an icon with a text label in list rows and nav.
- **Do** reserve green / amber / red for status; let colour mean something.

### Don't:
- **Don't** look like the **official school portal** — no cramped, form-heavy, dated density. Space and hierarchy are the point.
- **Don't** look like a **generic template app** — the one-seed teal, the 粉圓 type, and the slate identity card are the identity; don't sand them off toward neutral default Material.
- **Don't** be **over-designed or flashy** — no gradient text, no glassmorphism, no decorative motion, no layered shadow stacks. Motion is spent only on the nav pill and state feedback.
- **Don't** feel like an **ad / marketing app** — no pop-ups, promos, or engagement bait.
- **Don't** hand-pick a second brand colour or a second typeface to make a screen "pop".
- **Don't** put one uniform radius (or an oversized 20px+ radius) on everything — that's the bubbly, generic reflex the Radius-by-Size Rule exists to prevent.
- **Don't** rely on faux-bold for emphasis; it renders thin and uneven on the single-weight font.
- **Don't** use raw `Colors.grey` / hardcoded hex for surfaces or text — use Material 3 roles so the one-seed system stays intact.
